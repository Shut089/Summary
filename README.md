# План автоматизации тестирования: запись на профессию «Инженер по тестированию» (Netology)

## 1. Перечень автоматизируемых сценариев

### A. Переход на страницу профессии с главной страницы
1. Переход через меню __«Каталог курсов»__ → направление/раздел, связанный с тестированием → выбор профессии «Инженер по тестированию». 
2. Переход через контент на главной странице (промо-блоки/подборки/карточки профессий) → «Инженер по тестированию».
- Проверки: страница открылась; заголовок/хлебные крошки/URL соответствуют странице профессии.

> Ожидаемая целевая страница профессии (как минимум один из вариантов): `netology.ru/programs/qa`.  

### B. Переход к форме записи со страницы профессии
1. Скролл страницы профессии до формы записи:
- Проверки: форма видима; все обязательные поля присутствуют; блок не перекрыт поп-апами/куки-баннером.
2. Нажатие кнопки __«Записаться»__ на странице профессии:
- Проверки: произошёл переход/проскролл к форме; форма видима; фокус в первом поле формы.

### C. Заполнение формы (без отправки)
1. Позитивный сценарий заполнения валидными данными __без нажатия “Записаться”__:
- Ввод валидных значений во все обязательные поля.
- Отметка чекбокса согласия (если он обязателен).
- Проверки: нет сообщений об ошибках; поля не подсвечены 
как ошибочные; кнопка отправки остаётся __не нажатой__.
2. Валидации полей (набор коротких негативных сценариев, тоже без отправки):
- Пустые обязательные поля (по одному): имя/телефон/email (или фактически присутствующие обязательные поля).
- Невалидный телефон (буквы, слишком короткий/длинный, неверный формат).
- Невалидный email (без @, без домена, пробелы).
- Имя: цифры/спецсимволы (если есть ограничения), слишком короткое/слишком длинное.
- Не отмечен чекбокс согласия (если обязателен).
- Проверки: корректный текст ошибки; ошибка исчезает после исправления; ошибки не “залипают”.
3. UX/устойчивость:
- Повторное заполнение (ввод → очистка → ввод) без деградации.
- Переключение вкладок/перезагрузка страницы — корректное состояние формы (без отправки).
- Проверка, что любые действия автотеста не приводят к реальной отправке заявки.

## 2. Перечень используемых инструментов с обоснованием выбора
- __Java + Selenide__ — быстрые и стабильные UI-проверки, лаконичные селекторы и ожидания, удобно для динамических страниц (у вас уже есть опыт с Selenide).
- __JUnit 5__ — стандартный раннер тестов, теги/параметризация.
- __Gradle__ — сборка и запуск в CI.
- __Allure Report__ (опционально, но желательно) — понятные отчёты: шаги, скриншоты, источники данных.
- __JavaFaker__ — генерация валидных тестовых данных (имя, почта), с гарантией, что email/телефон “тестовые” и не приводят к реальным коммуникациям.
- __Page Object__ — поддерживаемость при изменениях верстки.
- __GitHub Actions__ (опционально) — прогон “smoke”, но только после появления разрешений/тестового стенда.

Ключевая инженерная оговорка: в проект закладывается “предохранитель” — отдельный флаг конфигурации, при котором тесты никогда не кликают по финальной кнопке отправки (и это отдельно проверяется код-ревью/линтами/тестами на уровне фреймворка).

## 3. Перечень необходимых разрешений, данных и доступов

Так как это реальный сайт и реальная форма заявки:

1. __Письменное разрешение__ от владельца ресурса (Нетология) на автоматизированное тестирование публичного сайта, с указанием допустимой нагрузки и временных окон.
2. __Тестовый стенд:__
- либо отдельный домен/окружение,
- либо feature-flag, переводящий форму в режим “не отправлять, а логировать”.
3. Доступ/согласование по антибот-механизмам:
- whitelist IP/UA,
- тестовые ключи CAPTCHA (если есть) или отключение на тестовом стенде.
4.	__Тестовые данные:__
- тестовые телефоны/почты (не принадлежащие реальным людям),
- правила валидации (длина/формат), если они отличаются от “очевидных”.
5.	__Доступ к логам/метрикам__ тестового стенда для расследования падений.
6.	Согласование, что __страница профессии__ может быть доступна без авторизации (на некоторых страницах у Нетологии встречаются экраны “нет доступа”, что влияет на тестируемость).  ￼

## 4. Перечень и описание возможных рисков при автоматизации
- __Юридический/этический риск:__ любая отправка формы создаёт реальную заявку и нагрузку на отдел продаж/поддержки (поэтому в плане — строго “без submit”, и нужен тестовый стенд/разрешение).
- __Нестабильность UI:__ динамические блоки, A/B-эксперименты, персонализация, меняющиеся селекторы.
- __Антибот/капча/рейтконтроль:__ автотесты могут блокироваться или, хуже, приводить к бану.
- __Поп-апы/куки-баннеры/виджеты:__ перекрывают элементы и дают флейки.
- __Зависимость от сторонних сервисов:__  падения не по вине продукта.
- __Хрупкие проверки контента:__ тексты и цены/акции часто меняются; тесты должны проверять структуру и состояние формы, а не маркетинговые формулировки.
- __Риск утечки тестовых данных__ (если случайно использовать реальные телефоны/почты) — нужен строгий контроль тест-данных и запрет на submit.

## 5. Перечень необходимых специалистов для автоматизации
- __QA Automation Engineer__ — разработка автотестов, инфраструктура фреймворка.
- __QA Analyst / Manual QA__ — проработка сценариев, критерии приемки, модель данных, негативные проверки.
- __Разработчик (frontend/backend)__ — консультации по валидациям/селекторным якорям.
- __Представитель владельца продукта__ — чтобы согласовать, что именно считаем формой записи и какие варианты перехода поддерживаются.
- __Юрист__ — согласование правил тестирования публичного сайта и обработки данных.

## 6. Интервальная оценка с учётом рисков (в часах)

Оценка дана диапазонами, потому что без подтверждённого тестового стенда и без данных о защите формы разброс существенно выше:
1.	Подготовка репозитория, шаблон проекта, конфиги: __4–8 ч__
2.	Page Object модель для главной → каталог → профессия → форма: __6–14 ч__
3.	Сценарии перехода к форме (скролл + кнопка “Записаться”): __3–6 ч__
4.	Сценарии заполнения формы (позитив + набор негативов): __8–18 ч__
5.	Устойчивость (работа с баннерами/поп-апами, ожидания, стабильные селекторы): __6–16 ч__
6.	Отчёты (Allure), артефакты (скрины/видео), документация README: __3–8 ч__
7.	CI (при наличии разрешений): __3–10 ч__

__Итого: 33–80 часов__, где верхняя граница — это цена рисков (отсутствие тестового стенда, высокая динамика страницы).
